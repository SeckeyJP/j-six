# 論点2：設計書逆生成の限界と対策

## — コードは「What」と「How」を語るが「Why」と「Why Not」を語らない —

---

## 1. 問題の本質

J-SIX プロセスでは、設計書を事前に書くのではなく、実装済みコードから設計書を **逆生成** する。これは設計書とコードの乖離を原理的に排除する画期的なアプローチだが、根本的な限界がある。

> **コードには「何をどう実装したか」は刻まれているが、「なぜそうしたか」「なぜ他の方法を採らなかったか」は刻まれていない。**

日本のSI案件で求められる設計書には、この「Why」と「Why Not」が含まれることが多い。ここにギャップが生じる。

---

## 2. 情報の4象限：コードから復元できるもの・できないもの

### 2.1 4象限マトリクス

```
                    コードに存在する        コードに存在しない
                  ┌─────────────────────┬─────────────────────┐
                  │                     │                     │
  構造的          │  ◎ 完全復元可能      │  ▲ 部分復元可能      │
  （静的情報）    │                     │                     │
                  │  ・クラス/関数構造   │  ・システム全体の     │
                  │  ・DB スキーマ       │    配置構成           │
                  │  ・API エンドポイント│  ・外部IF の相手先    │
                  │  ・依存関係         │    仕様               │
                  │  ・データ型/制約    │  ・非機能要件の       │
                  │                     │    数値目標           │
                  │                     │                     │
                  ├─────────────────────┼─────────────────────┤
                  │                     │                     │
  意図的          │  ○ 推測可能          │  × 復元不可能        │
  （暗黙知）      │                     │                     │
                  │  ・処理の振る舞い   │  ・設計判断の理由     │
                  │    （テストから）    │  ・却下した代替案     │
                  │  ・エラーハンドリング│  ・業務上の背景知識   │
                  │    方針（パターン） │  ・将来の拡張計画     │
                  │  ・命名規則         │  ・ステークホルダー   │
                  │    （コードから）    │    間の合意事項       │
                  │                     │  ・制約の根拠         │
                  │                     │                     │
                  └─────────────────────┴─────────────────────┘
```

### 2.2 各象限の詳細

#### ◎ 完全復元可能（コードに存在する構造的情報）

| 情報 | 復元方法 | 対応する設計書 |
|---|---|---|
| クラス/関数の構造 | ソースコード解析 | 詳細設計書（プログラム構造） |
| DB スキーマ（テーブル、カラム、型、制約） | DDL / マイグレーション解析 | DB設計書 |
| API エンドポイント（URL、メソッド、パラメータ） | OpenAPI / ルーティング解析 | IF設計書 |
| 依存関係（ライブラリ、モジュール間） | package.json / import 解析 | アーキテクチャ設計書 |
| データバリデーションルール | コード解析 | 入出力設計書 |
| 画面構成・遷移 | フロントエンドコード解析 | 画面設計書 |

**結論**: この領域は CC による逆生成が **完全に機能する**。現時点でも富士通等が「設計書リバースエンジニアリング」サービスを提供しており（Fujitsu, 2025.02）、手動比50%の効率化を報告している。

#### ○ 推測可能（コードに存在する意図的情報）

| 情報 | 推測方法 | 精度 |
|---|---|---|
| 処理の振る舞い | テストコードから推測 | 高（テストが充実していれば） |
| エラーハンドリング方針 | コードパターンの分析 | 中〜高 |
| 命名規則・コーディング規約 | コードの統計的分析 | 高 |
| セキュリティ対策方針 | 実装パターンの分析 | 中 |

**結論**: CC はパターン分析に優れるため、コードから暗黙のルールを抽出する能力は高い。ただし「推測」であるため、人間の確認が必要。

#### ▲ 部分復元可能（コードに存在しない構造的情報）

| 情報 | 部分復元の方法 | 不足する部分 |
|---|---|---|
| システム構成図（物理配置） | docker-compose / IaC から推測 | 本番環境固有の構成 |
| 外部IF の相手先仕様 | APIクライアントコードから推測 | 相手先の制約・SLA |
| 非機能要件（性能目標） | テストコード / 設定値から推測 | 要求の根拠・算出根拠 |
| 運用手順 | スクリプト / 設定から推測 | 運用判断基準 |

**結論**: IaC（Infrastructure as Code）やCI/CD設定が充実していれば復元率は上がるが、「本番環境固有の情報」や「要件の根拠」は復元不可。

#### × 復元不可能（コードに存在しない意図的情報）

| 情報 | なぜコードに残らないか | 影響 |
|---|---|---|
| **設計判断の理由（Why）** | コードは「What」しか表現しない | 保守担当者が意図を理解できない |
| **却下した代替案（Why Not）** | 採用されなかった案はコードに痕跡を残さない | 同じ検討を繰り返すリスク |
| **業務上の背景知識** | コードは技術的実装であり業務知識は含まない | 業務変更時の影響分析が困難 |
| **将来の拡張計画** | 未実装の計画はコードに存在しない | 拡張時に設計意図と衝突するリスク |
| **ステークホルダー間の合意事項** | 政治的・組織的な判断はコードに反映されない | 変更要求時の判断根拠が不明 |
| **制約の根拠** | 「なぜこの制約があるか」はコードに残らない | 制約の緩和可否が判断できない |
| **検討時の前提条件** | 当時の技術的・ビジネス的前提はコードに反映されない | 前提が変わった時の再検討が困難 |

**結論**: この領域は逆生成では **原理的に復元不可能** であり、別の仕組みが必要。

## 3. 対策：3層ドキュメント戦略

復元不可能な情報を補完するため、**3層のドキュメント戦略** を提案する。

```
┌────────────────────────────────────────────────────────────┐
│ 第1層：Spec（仕様書）— Phase 1-2 で人間が作成              │
│                                                            │
│  「何を作るか」「なぜ作るか」を記述                         │
│   ・要求Spec（業務背景・目的・スコープ・制約）             │
│   ・Design Spec（アーキテクチャ方針・技術選定理由）         │
│   ※ J-SIX プロセスの Phase 1-2 で作成済み               │
│                                                            │
├────────────────────────────────────────────────────────────┤
│ 第2層：ADR（Architecture Decision Records）— 随時記録      │
│                                                            │
│  「なぜそう判断したか」「他の選択肢は何だったか」を記録     │
│   ・設計判断とその理由                                     │
│   ・検討した代替案と却下理由                               │
│   ・前提条件・制約の根拠                                   │
│   ・将来の拡張計画・注意事項                               │
│   ※ Phase 2-4 で判断が発生するたびに CC が自動記録        │
│                                                            │
├────────────────────────────────────────────────────────────┤
│ 第3層：設計書（逆生成ドキュメント）— Phase 6 で自動生成    │
│                                                            │
│  「何をどう実装したか」をコードから自動生成                 │
│   ・基本設計書・詳細設計書・IF設計書・DB設計書             │
│   ・テスト仕様書・テスト結果                               │
│   ※ CC がコードから逆生成。従来フォーマット対応            │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 3.1 第1層：Spec（仕様書）

J-SIX プロセスの Phase 1（要求の合意）と Phase 2（技術設計）で作成する Spec は、コードには残らない **業務背景・目的・制約** を記録する。

**重要**: Spec は J-SIX プロセスにおいて **事前に作成する唯一のドキュメント** である。従来の「設計書を先に書く」文化との違いは、Spec が「実装の手順書」ではなく「意図の記録」である点。

#### Spec に含めるべき内容

```markdown
# 要求 Spec テンプレート

## 1. 業務背景
- 現行業務の課題
- 本システムで解決したいこと
- ステークホルダーと期待

## 2. スコープ
- 対象業務範囲
- 対象外の明示

## 3. 業務要件
- ユースケース一覧
- 業務ルール
- 非機能要件（性能・可用性・セキュリティ）

## 4. 制約条件
- 技術的制約とその根拠
- 組織的制約（既存システム連携、運用体制等）
- 法規制・コンプライアンス要件

## 5. 前提条件
- 本Specが前提としている状況
- 前提が変わった場合の影響
```

### 3.2 第2層：ADR（Architecture Decision Records）

**これが本論点における最大の提案**。ADR は設計書逆生成の限界を補完する核心的な仕組みである。

#### なぜ ADR が必要か

コードからは「なぜ PostgreSQL を選んだか」は分からない。分かるのは「PostgreSQL を使っている」という事実だけ。ADR はこの **「Why」と「Why Not」** を記録する。

#### ADR の自動記録フロー

```
Phase 2（技術設計）中に設計判断が発生
    ↓
CC が判断を検出し、ADR テンプレートに沿って記録を提案
    ↓
人間が内容を確認・承認
    ↓
docs/adr/ に自動保存
    ↓
Phase 6（ドキュメント生成）で設計書に「設計根拠」として統合
```

#### CC による ADR 自動記録の実現方法

CC には設計判断を検出して ADR を自動提案する能力がある。以下のように CLAUDE.md と Hook で実装する。

**CLAUDE.md に記述するルール**:
```markdown
## ADR ルール
- アーキテクチャに影響する判断を行った場合、必ず ADR を作成すること
- ADR は docs/adr/NNNN-タイトル.md に保存
- テンプレートは docs/adr/template.md に従う
- 以下の場合は ADR 必須：
  - 技術スタック/ライブラリの選定
  - データベース設計の重要な判断
  - API 設計方針の決定
  - セキュリティ方針の決定
  - 性能に影響する設計判断
  - 代替案を検討して却下した場合
```

**ADR テンプレート**:
```markdown
# ADR-NNNN: [タイトル]

## ステータス
[提案中 | 承認 | 非推奨 | 却下]

## 日付
YYYY-MM-DD

## コンテキスト（背景）
この判断が必要になった背景・課題は何か。

## 判断（Decision）
何を決定したか。

## 理由（Rationale）
なぜこの判断をしたか。

## 検討した代替案
| 代替案 | メリット | デメリット | 却下理由 |
|---|---|---|---|

## 影響（Consequences）
### ポジティブな影響
### ネガティブな影響（トレードオフ）
### 将来の注意事項

## 関連
- 関連する ADR: ADR-XXXX
- 関連する Spec: requirement-spec.md#セクション名
- 関連するコード: src/path/to/module
```

#### 実際の ADR 例

```markdown
# ADR-0003: セッション管理に Redis を採用

## ステータス
承認

## 日付
2026-04-15

## コンテキスト
Webアプリケーションのセッション管理方式を決定する必要がある。
スケールアウト時に複数サーバー間でセッションを共有する要件がある。
（要求Spec: requirement-spec.md#非機能要件-可用性）

## 判断
セッションストアとして Redis を採用する。

## 理由
- スケールアウト時のセッション共有が容易
- TTL によるセッション自動失効が組み込み
- 既存インフラに Redis クラスタが存在（運用コスト増なし）

## 検討した代替案
| 代替案 | メリット | デメリット | 却下理由 |
|---|---|---|---|
| DB（PostgreSQL） | 追加インフラ不要 | I/O負荷増、スケール困難 | 性能要件を満たさない |
| JWT（ステートレス） | サーバー側ストア不要 | トークン失効が困難、ペイロード肥大 | セキュリティ要件（即時セッション無効化）を満たさない |
| Memcached | 高速 | 永続化なし、データ型が限定 | Redis の方が機能が豊富で同等性能 |

## 影響
### ポジティブ
- 水平スケーリングが容易に
- セッションの即時無効化が可能
### ネガティブ
- Redis 障害時のフォールバック設計が必要
- Redis クラスタの監視運用が追加
### 将来の注意事項
- Redis 7.x 以降でのクラスタ構成変更に注意

## 関連
- ADR-0001: データベースに PostgreSQL を採用
- 要求Spec: requirement-spec.md#非機能要件-可用性
- 実装: src/infrastructure/session/redis-store.ts
```

### 3.3 第3層：設計書（逆生成ドキュメント）

Phase 6 で CC がコードから自動生成する。ここで重要なのは、第1層（Spec）と第2層（ADR）の情報を **統合** して設計書に反映する点。

#### 逆生成設計書の構成

```
【基本設計書】逆生成
├── 1. システム概要     ← Spec から引用
├── 2. アーキテクチャ   ← コードから生成 + ADR の設計根拠を注記
├── 3. 画面設計         ← フロントエンドコードから生成
├── 4. DB設計           ← DDL/マイグレーションから生成 + ADR の設計根拠
├── 5. IF設計           ← OpenAPI/コードから生成
├── 6. セキュリティ設計 ← コードから生成 + ADR の方針根拠
└── 7. 設計判断記録     ← ADR 一覧の組み込み

【詳細設計書】逆生成
├── 1. モジュール構成   ← コードから生成
├── 2. クラス/関数設計  ← コードから生成
├── 3. 処理フロー       ← コード + テストから生成
├── 4. エラー処理方針   ← コードパターンから生成
└── 5. 設計判断記録     ← 該当モジュールの ADR 組み込み
```

**最大のポイント**: 設計書の「設計根拠」欄に ADR の内容を自動挿入することで、「なぜそう設計したか」が設計書に組み込まれる。これは従来の手書き設計書でも実現が難しかった品質である。

## 4. 3層戦略による情報カバレッジ

### 4.1 情報カバレッジの比較

| 情報カテゴリ | 従来V字モデル | J-SIX（逆生成のみ） | J-SIX（3層戦略） |
|---|---|---|---|
| 構造的情報（What/How） | ○（設計書に記載） | ◎（コードから正確に生成） | ◎ |
| 業務背景（Why Business） | △（設計書に書くが形骸化しがち） | ×（コードに存在しない） | ◎（Spec で記録） |
| 設計判断理由（Why Technical） | △（暗黙知に留まることが多い） | ×（コードに存在しない） | ◎（ADR で記録） |
| 却下した代替案（Why Not） | ×（ほぼ記録されない） | ×（コードに存在しない） | ◎（ADR で記録） |
| 設計書⇔コードの整合性 | △（乖離が頻発） | ◎（原理的に一致） | ◎ |
| トレーサビリティ | ○（手動管理） | △（コード内の関連のみ） | ◎（Spec→ADR→コード→設計書の自動リンク） |

### 4.2 従来の設計書文化に対する優位性

**逆説的だが、3層戦略は従来の設計書よりも情報量が多い。**

従来の設計書文化では：
- 設計判断の理由は議事録やメールに散在し、設計書には記載されない
- 却下した代替案はほぼ記録されない（検討したこと自体が忘れられる）
- 設計書とコードが乖離し、「設計書に書いてあるが実装は違う」問題が常態化

3層戦略では：
- Spec で業務背景・制約を明示的に記録
- ADR で設計判断・代替案・トレードオフを体系的に記録
- 逆生成設計書により構造情報は常にコードと一致
- Spec → ADR → コード → 設計書の自動リンクでトレーサビリティ完備

## 5. 実装上の課題と解決策

### 5.1 ADR が増えすぎる問題

**課題**: プロジェクトが進むにつれ ADR が大量に蓄積され、管理が煩雑になる。

**解決策**:
- ADR に重要度（High/Medium/Low）を設定し、設計書に組み込むのは High のみ
- CLAUDE.md に「ADR 必須の判断基準」を明記し、些末な判断は ADR 化しない
- ADR のステータス管理（承認→非推奨→アーカイブ）で現役の ADR を絞り込み

### 5.2 ADR 記録のタイミング問題

**課題**: 設計判断の瞬間に ADR を書くのは面倒で、後回しにされがち。

**解決策**:
- CC が設計判断を検出した時点で ADR ドラフトを自動生成
- Hook で「アーキテクチャに影響するファイル変更時に ADR の有無をチェック」
- 人間は CC が生成した ADR ドラフトを確認・修正するだけ

### 5.3 設計書逆生成の品質問題

**課題**: CC が生成した設計書の日本語品質・フォーマット精度が不十分な可能性。

**解決策**:
- 設計書テンプレート（Excel/Word）を Skills として精密に定義
- 逆生成後の人間レビュー（Phase 6 の品質ゲート）
- 逆生成 Skill の継続的改善（プロジェクトフィードバックの蓄積）

### 5.4 顧客が「事前の設計書」を求める問題

**課題**: 契約上、実装前に設計書の提出・承認が求められるケースがある。

**解決策（段階的）**:
1. **短期**: Phase 2 の Design Spec + プロトタイプを「基本設計書相当」として提出。従来フォーマットへの変換は CC が行う
2. **中期**: 顧客と合意し、Spec + プロトタイプ + ADR を設計書の代替として受け入れてもらう
3. **長期**: 業界全体で「Spec + コード + テスト + ADR = 設計書」という認識を普及

## 6. 結論

### 6.1 設計書逆生成の限界は「ADR で補完可能」

コードから復元できない情報は主に「Why」と「Why Not」。この情報は ADR として設計判断の発生時に CC が自動記録することで補完できる。

### 6.2 3層戦略により従来の設計書文化を「超越」できる

Spec（Why Business） + ADR（Why Technical / Why Not） + 逆生成設計書（What / How） の3層により、従来の設計書よりも情報量が多く、かつコードとの整合性が保証された設計文書体系を実現できる。

### 6.3 CC Plugin への実装指針

| コンポーネント | 実装形態 | 優先度 |
|---|---|---|
| ADR テンプレート・管理 Skill | Skills + CLAUDE.md | 最高 |
| ADR 自動検出 Hook | Hooks（アーキテクチャ変更検出） | 高 |
| 設計書逆生成 Skill（基本設計書） | Skills | 高 |
| 設計書逆生成 Skill（詳細設計書） | Skills | 高 |
| 設計書逆生成 Skill（IF設計書） | Skills | 中 |
| 設計書逆生成 Skill（DB設計書） | Skills | 中 |
| Spec テンプレート | Skills | 高 |
| Spec→ADR→コード→設計書 トレーサビリティ Skill | Skills | 中 |

## 7. 次の論点への接続

本論点の結論（3層戦略）は、論点3「CC自律実行の範囲」と密接に関連する。

- Phase 4（TDD実装）で CC が自律実行する際、設計判断が発生した場合に ADR を自動記録する必要がある
- CC 自律実行の範囲を決めることは、「どのレベルの判断を人間に委ね、どのレベルを CC に任せるか」を決めることであり、ADR の粒度にも影響する
