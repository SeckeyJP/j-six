# 論点4：段階的移行パス（V字モデル → J-SIX）

## — 既存プロジェクトを止めずに移行する現実的なロードマップ —

---

## 1. 移行の大前提

### 1.1 「ビッグバン移行」はしない

日本のSI現場で「明日からJ-SIXで」は不可能であり、やるべきでもない。理由は3つ。

1. **進行中のプロジェクトがある** — 既存のV字モデルで走っているプロジェクトを中断できない
2. **組織的な学習が必要** — CC操作スキル、TDD文化、Spec駆動の考え方は一朝一夕には浸透しない
3. **顧客・経営層の信頼が必要** — 定量的な効果を示しながら段階的に移行することで合意を得る

### 1.2 移行の基本戦略

> **既存プロセスの「内側から」CCを浸透させ、結果として J-SIX に到達する**

V字モデルを「廃止」するのではなく、各工程で CC 活用を深めていくうちに、プロセスが自然に J-SIX に「進化」する。

### 1.3 これまでの論点との関係

| 論点 | 結論 | 移行パスへの影響 |
|---|---|---|
| **論点1**: SDD vs 独自 | CC ネイティブ + 日本品質レイヤー | Plugin として段階的に導入。全機能一括ではなく部分導入可能 |
| **論点2**: 設計書逆生成 | 3層戦略（Spec + ADR + 逆生成） | ADR を先に導入。設計書逆生成は後期に導入 |
| **論点3**: 自律実行範囲 | L1→L2→L3→L4 の段階的引き上げ | 移行ステージと自律度の引き上げを同期させる |

## 2. 移行の4ステージ

```
Stage 0          Stage 1            Stage 2            Stage 3
従来V字モデル  → CC補助V字モデル  → CC駆動ハイブリッド → J-SIX
                  (3ヶ月)           (3-6ヶ月)           (6ヶ月〜)

自律度: L0        L1-L2              L2-L3              L3-L4
プロセス: V字     V字+CC支援         SDD+V字成果物互換   J-SIX(Phase0-6)
設計書: 事前作成   事前作成+CC支援    並行（Spec+従来）   逆生成（3層戦略）
テスト: 実装後     実装後+CC生成UT    TDD（部分）         TDD（全面）
品質: レビュー会議  CC事前+レビュー    CC相互+サンプリング  自動+サンプリング
```

## 3. Stage 0: 現状（従来V字モデル）

多くの日本SI案件の現在地。移行のベースラインとして定義する。

```
要件定義 → 基本設計 → 詳細設計 → 実装 → UT → IT → ST
  ↓          ↓          ↓        ↓     ↓    ↓    ↓
要件定義書  基本設計書  詳細設計書 コード  UT結果 IT結果 ST結果
(Word/Excel) (Excel方眼紙)  ...
```

**特徴**:
- 設計書を事前に作成し、レビュー会議で承認
- 実装は設計書に基づいて人間が行う
- テストは実装後に作成
- CC は未使用 or 個人レベルでの非公式利用

---

## 4. Stage 1: CC補助V字モデル（3ヶ月）

### 4.1 目的

- **V字モデルのプロセスは変えない**まま、各工程で CC を補助的に活用
- CC 活用のスキルを組織的に蓄積
- 効果を定量的に計測し、次ステージへの根拠を作る

### 4.2 導入する CC 活用（自律度 L1-L2）

| 工程 | CC の活用方法 | 従来プロセスとの関係 | 自律度 |
|---|---|---|---|
| 要件定義 | 要件の曖昧性チェック、業務フロー図の下書き生成 | 従来の要件定義プロセスに追加 | L1 |
| 基本設計 | 設計書ドラフトの生成支援、プロトタイプ生成 | 人間が設計、CC がドラフト化を支援 | L1 |
| 詳細設計 | 詳細設計書のドラフト生成 | 人間が設計、CC が記述を支援 | L1 |
| **実装** | **コード生成 + 人間レビュー** | **CC が生成、人間が承認（L2）** | **L2** |
| **UT** | **テストコード自動生成** | **CC が生成、人間がレビュー** | **L2** |
| IT/ST | テストケース生成支援 | 設計書からテストケースの下書き | L1 |

### 4.3 この段階で導入するもの

```
導入物                          目的
────────────────────────────────────────────────
CLAUDE.md（基本版）           → CC へのプロジェクトルール伝達
  ・コーディング規約
  ・ディレクトリ構成
  ・ビルド/テストコマンド

効果計測の仕組み              → 次ステージへの意思決定根拠
  ・CC 活用/非活用の工数比較
  ・CC 生成コードの品質メトリクス
  ・開発者の満足度調査
```

### 4.4 この段階では導入しないもの

- TDD（既存のテストプロセスを維持）
- 設計書逆生成（従来通り事前作成）
- ADR（まだ早い）
- サブエージェント/Agent Teams（単一セッションのみ）
- Hooks（手動チェックのまま）

### 4.5 Stage 1 の完了基準（次ステージへの移行判定）

| 基準 | 閾値 |
|---|---|
| CC 活用による実装工数削減率 | 20%以上 |
| CC 生成コードのバグ密度 | 人間コードの2倍以内 |
| CC 生成テストのカバレッジ | 60%以上 |
| 開発者の CC 操作習熟度 | チーム全員が日常的に使用 |
| CLAUDE.md の充実度 | コーディング規約・ビルドコマンドが網羅 |

## 5. Stage 2: CC駆動ハイブリッド（3-6ヶ月）

### 5.1 目的

- **V字モデルの骨格は維持しつつ、内部のプロセスを CC 駆動に置換**
- TDD を部分的に導入（新規モジュールから）
- ADR の運用を開始
- 設計書の「二重作成」（従来方式 + 逆生成の試行）

### 5.2 プロセスの変化

```
【Stage 1】                      【Stage 2】
要件定義 → 基本設計 → ...        要件定義 → 基本設計 → ...
                                      ↓          ↓
従来プロセス + CC 補助          Spec作成     Design Spec 作成
                                 (CC支援)     (CC支援)
                                      ↓          ↓
                                 従来設計書    従来設計書
                                  も並行作成    も並行作成（移行期）
```

### 5.3 導入する CC 活用（自律度 L2-L3）

| 工程 | CC の活用方法 | 変化のポイント | 自律度 |
|---|---|---|---|
| 要件定義 | **Spec 作成の開始**（従来の要件定義書と並行） | Spec が実装を駆動する起点に | L2 |
| 基本設計 | **Design Spec + プロトタイプ駆動**（従来設計書も並行） | 「紙の設計書」→「動くモック」でレビュー | L2 |
| 詳細設計 | **タスク分解（CC提案→人間承認）** | 詳細設計書→タスクへの置換開始 | L2 |
| 実装 | **新規モジュールで TDD 導入** | テストファースト、サブエージェント分離 | **L3** |
| UT | **CC 自律実行（新規モジュール）** | TDD サイクル自律実行の開始 | **L3** |
| IT/ST | Spec からテストケース自動生成 | テストケース作成の自動化 | L2 |
| （新規）| **ADR の運用開始** | 設計判断の記録習慣化 | L2 |

### 5.4 この段階で導入するもの

```
導入物                              目的
──────────────────────────────────────────────────
CLAUDE.md（拡充版）               → 設計方針・品質基準を追加
  ・設計方針（DB/API/画面）
  ・品質基準（カバレッジ目標等）
  ・ADR ルール
  ・禁止事項・制約

Spec テンプレート                  → Spec 駆動開発の基盤
  ・要求 Spec テンプレート
  ・Design Spec テンプレート

ADR テンプレート + 運用ルール      → 設計判断の形式知化
  ・テンプレート
  ・CLAUDE.md への ADR ルール記載
  ・CC による ADR 自動提案

Hooks（基本セット）               → 品質ガードレールの自動化
  ・pre-commit: lint + カバレッジ
  ・post-tool-use: ADR チェック

サブエージェント（TDD用）         → TDD サイクルの分離
  ・Red Agent / Green Agent / Refactor Agent

設計書逆生成の試行                → 従来設計書と逆生成の品質比較
  ・一部モジュールで逆生成を試行
  ・従来設計書と並べて品質比較
```

### 5.5 「二重作成」の意義

Stage 2 では、一部のモジュールについて **従来方式の設計書** と **逆生成の設計書** を両方作成する。目的は：

1. 逆生成設計書の品質を定量的に評価する
2. 顧客・QA部門に逆生成設計書の実物を見せ、受容性を確認する
3. 「従来設計書と同等以上」のエビデンスを蓄積する

### 5.6 Stage 2 の完了基準

| 基準 | 閾値 |
|---|---|
| TDD 導入モジュールの品質 | 非TDDモジュールと同等以上 |
| CC 自律実行（L3）の安定性 | エスカレーション率 20%以下 |
| 逆生成設計書の品質 | 従来設計書と同等（レビューア評価） |
| ADR の運用定着 | 主要な設計判断の80%以上が ADR 化 |
| Spec → タスク → 実装 のフロー | 新規モジュールで安定稼働 |
| 開発工数削減率 | 40%以上（Stage 0 比） |

## 6. Stage 3: J-SIX（6ヶ月〜）

### 6.1 目的

- **J-SIX プロセス（Phase 0-6）の全面適用**
- 設計書は逆生成に完全移行（従来方式の事前作成を廃止）
- TDD の全面適用
- L3-L4 自律実行の本格運用

### 6.2 プロセスの全体像（J-SIX Phase 0-6）

```
Phase 0: CLAUDE.md 策定                      ← L1
Phase 1: 要求の合意（Spec 策定）             ← L1-L2
Phase 2: 技術設計（Design Spec + プロトタイプ）← L2
Phase 3: タスク分解                           ← L2
Phase 4: TDD 実装                             ← L3-L4 ★生産性の源泉
Phase 5: 品質検証                             ← L2-L3
Phase 6: ドキュメント生成（3層戦略）          ← L3
```

### 6.3 Stage 2 からの変化点

| 観点 | Stage 2 | Stage 3（J-SIX） |
|---|---|---|
| 設計書 | 従来方式 + 逆生成の二重作成 | **逆生成のみ**（3層戦略） |
| TDD | 新規モジュールのみ | **全モジュール** |
| 自律度 | L2-L3（部分自律） | **L3-L4（全面自律）** |
| 並列実行 | 単一セッション中心 | **Agent Teams + git worktree** |
| 品質チェック | Hooks（基本） | **Hooks（完全セット）+ 自動品質ダッシュボード** |
| Spec | 従来設計書と並行 | **Spec が唯一の上流成果物** |

### 6.4 この段階で導入するもの

```
導入物                                目的
────────────────────────────────────────────────────
Agent Teams                         → 並列タスク実行の本格化
git worktree 運用                   → 並列作業空間の確保
Writer/Reviewer パターン            → CC 相互レビューの本格運用
品質ダッシュボード                  → リアルタイム品質監視
設計書逆生成 Skills（完全版）       → 全設計書の逆生成対応
  ・基本設計書 / 詳細設計書
  ・IF設計書 / DB設計書
  ・テスト仕様書
Plugin パッケージ化                 → 他プロジェクトへの展開
/loop による継続的品質監視          → 定期的な品質チェック自動実行
```

### 6.5 顧客への説明と合意

Stage 3 への移行には顧客の合意が必要。以下のエビデンスを提示する。

| 提示するエビデンス | 説得力のポイント |
|---|---|
| Stage 1-2 の工数削減実績 | 「40%以上の工数削減」の実績 |
| 逆生成設計書の品質比較結果 | 「従来設計書と同等以上」のエビデンス |
| 品質メトリクスの推移 | バグ密度・カバレッジの改善トレンド |
| ADR による設計判断の可視化 | 「従来より設計根拠が明確」の実証 |
| Spec + プロトタイプの有効性 | 「紙の設計書よりレビューが効率的」の実感 |

## 8. 移行を阻む壁と対策

### 8.1 組織的な壁

| 壁 | 対策 |
|---|---|
| **「V字モデルで問題ない」という保守意識** | Stage 1 の効果を定量的に示す。「問題ない」ではなく「もっと良くなる」の訴求 |
| **CC に仕事を奪われる不安** | SE/PGの役割が「AI開発オーケストレーター」に進化することを明示。スキルアップの機会として位置づけ |
| **経営層のROI懸念** | Stage 1 の投資対効果を早期に算出。CC のサブスクリプション費用 vs 工数削減の比較 |
| **QA部門の抵抗** | QAの役割を「テスト戦略策定 + サンプリング検証」に再定義。探索的テストの重要性を強調 |

### 8.2 技術的な壁

| 壁 | 対策 |
|---|---|
| **TDD の経験がない** | Stage 2 で新規モジュールから段階的に導入。CC が TDD サイクルを主導するため、学習コストは低い |
| **レガシーコードへの適用** | レガシーコードは Stage 1 の L1-L2 で対応。リファクタリング時に段階的に TDD 化 |
| **CLAUDE.md の書き方がわからない** | テンプレートと記述ガイドを Plugin で提供。/init コマンドで初期生成 |
| **セキュリティ懸念（コードの外部送信）** | Anthropic API の利用規約を確認。Enterprise プラン + VPC 環境の検討。機密情報マスキングルール |

### 8.3 顧客との壁

| 壁 | 対策 |
|---|---|
| **「設計書を先に見せろ」という要求** | Stage 2 で Spec + プロトタイプの有効性を実証。「動くもの」でレビューする文化を提案 |
| **「AI生成コードは信頼できない」** | 品質メトリクスの継続的提示。「人間が全てレビュー」の保証（Stage 1-2） |
| **契約上の成果物要件** | 逆生成設計書が契約要件を満たすことを Stage 2 で実証。必要に応じて契約内容の更新を交渉 |

## 9. 移行の投資対効果（ROI）試算

※ 以下は仮定条件に基づく **試算** であり、実際の効果はプロジェクト特性により異なる。

### 9.1 前提条件

| 項目 | 値 |
|---|---|
| プロジェクト規模 | 中規模（10人月相当） |
| 開発者数 | 5名 |
| CC サブスクリプション | Max 5x（$100/月/人） |
| 開発者月額人件費 | 80万円/人 |

### 9.2 ステージ別 ROI

| ステージ | CC費用（月額） | 工数削減率 | 削減額（月額） | 月次ROI |
|---|---|---|---|---|
| Stage 1 | 約7.5万円（5人×$100） | 25% | 100万円（5人×80万円×25%） | **+92.5万円** |
| Stage 2 | 約7.5万円 | 45% | 180万円 | **+172.5万円** |
| Stage 3 | 約7.5万円 | 65% | 260万円 | **+252.5万円** |

※ API 利用料が別途発生する場合は追加コスト。ただし工数削減効果が圧倒的に大きい。
※ Stage 1 の導入初期は学習コストにより効果が低く、月を追って改善する想定。

### 9.3 経営層への提示ポイント

- **Stage 1 の投資回収は即月** — CC費用7.5万円に対し、25%の工数削減で100万円相当の効果
- **リスクは限定的** — Stage 1 は既存プロセスを変えないため、失敗してもダウンサイドが小さい
- **段階的投資** — 効果を確認しながら次ステージに進むため、大型投資リスクなし

## 10. 結論

### 10.1 移行パスの要約

1. **Stage 1（3ヶ月）**: V字モデルを変えずに CC を補助的に導入。CLAUDE.md 基本版を整備。効果を計測
2. **Stage 2（3-6ヶ月）**: SDD の要素を段階的に導入。TDD を新規モジュールから開始。ADR 運用開始。設計書の二重作成で逆生成の品質を実証
3. **Stage 3（6ヶ月〜）**: J-SIX プロセスに全面移行。設計書は逆生成のみ（3層戦略）。L3-L4 自律実行の本格運用

### 10.2 成功の鍵

- **定量的な効果計測** — 各ステージで効果を数値化し、次ステージへの根拠とする
- **段階的な信頼構築** — 組織・顧客・QA部門の信頼を段階的に獲得する
- **Plugin による導入容易性** — ワンコマンドで必要なツール一式を導入できる形態

### 10.3 全4論点の統合

これで4つの論点すべてが議論完了。全体像は以下の通り。

| 論点 | 結論 | J-SIX プロセスへの反映 |
|---|---|---|
| **1. SDD vs 独自** | CC ネイティブ + 日本品質レイヤー | CC Plugin として実装・配布 |
| **2. 設計書逆生成** | 3層戦略（Spec + ADR + 逆生成） | Phase 1-2 で Spec、随時 ADR、Phase 6 で逆生成 |
| **3. 自律実行範囲** | L3-L4 で TDD 実装（ガードレール付き） | Phase 4 が生産性の源泉 |
| **4. 段階的移行** | 3ステージ（補助→ハイブリッド→Full） | 既存案件を止めずに移行可能 |
