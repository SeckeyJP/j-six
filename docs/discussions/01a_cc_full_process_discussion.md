# J-SIX：AI ネイティブ開発プロセスの探求

## — 世界標準の AI ネイティブ開発と日本品質基準の融合 —

---

## 0. 前版への自己批判

前版（01_process_overview.md）は、従来のV字モデルにCCを「貼り付けた」だけであった。
これは **既存プロセスの改善** であり、**CCをフル活用したプロセスの再設計** ではない。

本ドキュメントでは、以下の問いに正面から向き合う。

> **「CCの能力を前提としたとき、品質を担保しつつ最大の生産性を発揮するプロセスとは何か？」**

---

## 1. 従来プロセスの前提が崩壊している

### 1.1 V字モデルが成立した前提条件

日本のV字モデル（ウォーターフォール）は以下の前提のもとで合理的だった。

| 前提 | V字モデルでの対応 |
|---|---|
| **実装コストが高い** | 上流で詳細に設計し、手戻りを防ぐ |
| **人間が全行コードを書く** | 設計書で事前に論理を確定させる |
| **テスト作成が高コスト** | テスト仕様書を事前に準備して効率化 |
| **コミュニケーションコストが高い** | 設計書を「契約」として工程間の認識を統一 |
| **変更が困難** | ウォーターフォールで凍結点を設けて管理 |

### 1.2 CCによって崩壊する前提

| 崩壊する前提 | CC がもたらす現実 |
|---|---|
| 実装コストが高い | **実装コストは劇的に低下**。CCは数分で数百行のコードを生成する |
| 人間が全行コードを書く | **CCが大部分を生成**し、人間はレビュー・判断に集中 |
| テスト作成が高コスト | **テストコード自動生成**。TDDのサイクルがCC前提で回る |
| 変更が困難 | **変更コストが劇的に低下**。リファクタリングもCC任せ |

### 1.3 しかし崩壊しない前提

| 変わらない前提 | 理由 |
|---|---|
| **何を作るかの合意は人間が行う** | ビジネス要件・ステークホルダー合意はAIでは代替不可 |
| **アーキテクチャ判断は人間が行う** | 技術選定・非機能要件のトレードオフは文脈依存 |
| **品質の最終責任は人間にある** | CC生成コードは人間の約1.7倍のイシューを含むとの調査報告あり（CodeRabbit, 2025.12） |
| **トレーサビリティは必要** | 要件→設計→実装→テストの追跡可能性は監査・保守で不可欠 |
| **顧客との合意プロセスは必要** | 特に日本のSIモデルでは受発注関係がある |

## 2. 世界標準の AI ネイティブ開発プロセス

### 2.1 Spec-Driven Development（SDD）

現在、世界の先端で最も支持されている手法が **Spec-Driven Development** である。

**核心思想**: 仕様書（Spec）を「第一級の実行可能な成果物」として扱い、仕様から直接実装を生成する。

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  1. SPECIFY  │────▶│   2. DESIGN  │────▶│   3. TASKS   │────▶│ 4. IMPLEMENT │
│              │     │              │     │              │     │              │
│ 何を・なぜ   │     │ どう作るか   │     │ 分割・依存   │     │ CC自律実行   │
│ 人間が承認   │     │ 人間が承認   │     │ 人間が承認   │     │ テストで検証 │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
      ▲ 人間                ▲ 人間                ▲ 人間              ▲ 自動
      │ ゲート              │ ゲート              │ ゲート            │ ゲート
```

**V字モデルとの本質的な違い**:
- V字モデル: 各工程で「設計書」→「次工程に引き渡し」→「実装」→「テスト」と直列進行
- SDD: **Specが生きた文書**として実装・テストを直接駆動し、乖離を許さない

### 2.2 主要フレームワークの比較

| フレームワーク | 特徴 | 規模適性 | CC統合度 |
|---|---|---|---|
| **BMAD Method** | 21の専門AIエージェント、アジャイル統合、マルチドメイン対応 | 中〜大規模 | 高（CC推奨） |
| **GitHub Spec Kit** | GitHub公式、CLI提供、軽量 | 小〜中規模 | 中 |
| **Superpowers** | TDD中心、7フェーズ、Anthropic公式マーケットプレイス入り | 小〜中規模 | 最高 |
| **SPARC** | 5フェーズ（仕様→擬似コード→設計→改善→完成）、並列実行 | 中規模 | 高 |
| **cc-sdd** | Kiro風コマンド、8エージェント対応、13言語 | 小〜大規模 | 高 |

### 2.3 共通する設計原則

世界標準の各フレームワークに共通する原則がある。

1. **Spec First（仕様駆動）**: コードの前に仕様。仕様が Source of Truth
2. **Phase Gates（フェーズゲート）**: 個々の編集ではなく、フェーズ完了時に人間が承認
3. **TDD（テスト駆動）**: テストを先に書き、CCに実装させる。最も品質が安定するパターン
4. **Parallel Agents（並列エージェント）**: Writer/Reviewer分離、サブエージェントによるコンテキスト分離
5. **Living Documentation（生きたドキュメント）**: 設計書はコードと常に同期
6. **CLAUDE.md as Constitution（プロジェクト憲法）**: 全ルール・規約・判断基準を一箇所に集約

### 3.2 従来V字モデル・SDD・J-SIXの本質的な違い

| 観点 | V字モデル（従来日本） | SDD（世界標準） | J-SIX（本提案） |
|---|---|---|---|
| **Source of Truth** | 設計書（Excel/Word） | Spec（Markdown） | Spec + コード + テスト（三位一体） |
| **設計書の位置づけ** | 実装の前に書く「設計図」 | 実装を駆動する「仕様」 | 実装後にコードから逆生成する「納品物」 |
| **実装者** | 人間（SE/PG） | CCが主、人間がレビュー | CC自律実行（TDDサイクル） |
| **品質保証の方法** | レビュー会議 + テスト | Phase Gate + テスト | 自動テスト + CC相互レビュー + 人間サンプリング |
| **テストの位置づけ** | 実装後に書く | Spec後に書く | **実装前に書く（TDD）** |
| **並列性** | 基本的に直列 | タスク単位で並列 | **git worktreeで大規模並列** |
| **変更への耐性** | 低い（凍結管理） | 中（Spec更新で追従） | **高い（TDD + 自動リグレッション）** |
| **日本の設計書文化** | ネイティブに対応 | 追加対応が必要 | **Phase 6で後工程生成** |
| **顧客承認** | 各工程で承認 | Phase Gateで承認 | **Phase 1, 2 + Phase 6で承認** |

### 3.3 最大のパラダイムシフト：設計書は「事後生成」

従来の日本式開発プロセスにおける最も根深い信念は「設計書を先に書き、それに従って実装する」である。J-SIXプロセスではこれを逆転させる。

```
【従来】
  設計書 ──→ 実装 ──→ テスト ──→ (設計書との乖離が発覚) ──→ 手戻り

【J-SIX】
  Spec ──→ テスト ──→ 実装 ──→ 検証 ──→ 設計書の逆生成
                                            ↑
                                    コードが Source of Truth なので
                                    設計書との乖離が原理的に発生しない
```

**なぜこれが可能か**:
- CCがコードから設計書を自動生成できるため、「設計書を先に書く」動機が消失
- TDDにより品質はテストが保証するため、設計書の品質保証機能が不要
- 設計書は「実装を導くツール」から「コミュニケーション・納品物」に役割が変化

**なぜこれが日本で受け入れられるか**:
- 顧客に納品する設計書のフォーマット・内容は従来と同等を維持
- 設計書レビューも従来と同様に実施可能
- 「設計書がない」のではなく「設計書の作り方が変わる」だけ

## 4. 各Phaseの詳細と品質担保

### Phase 0: プロジェクト憲法策定（CLAUDE.md）

**従来の対応物**: 開発標準、コーディング規約、プロジェクト計画書

**J-SIXでの位置づけ**: プロジェクトの全ルールをCCが理解・実行可能な形式で記述した「憲法」。CCの全出力がこの文書に制約される。

**品質への影響**: CLAUDE.mdの品質がプロジェクト全体の品質を左右する。Anthropic社内の調査でも「CLAUDE.mdの充実度とCC出力品質は直接相関する」と報告されている。

### Phase 1: 要求の合意

**従来の対応物**: 要件定義工程

**J-SIXでの違い**:
- CCのインタビューモード（ask_user_question ツール）で、要件の曖昧性を自動検出
- 業務フローのプロトタイプを即時生成し、「動くもの」で顧客と合意形成
- ただし、**ステークホルダーとの合意形成そのものは人間が主導**

**品質担保**:
- 要求Specは顧客が承認（従来と同等のゲート）
- CCが生成した曖昧性指摘リストを活用してレビュー品質向上

### Phase 2: 技術設計

**従来の対応物**: 基本設計工程

**J-SIXでの違い**:
- アーキテクチャ判断は人間が行うが、**CCがその具現化を即時に行う**
- DB設計 → DDL → ER図 を一貫生成。Design Spec として Markdown で管理
- 画面設計 → 実動HTMLプロトタイプを即時生成（紙の設計書ではなく動くモック）
- 技術選定の調査をCCサブエージェントが並列実行

**品質担保**:
- 設計レビューは実施するが、レビュー対象が「紙の設計書」から「動くプロトタイプ」に変化
- これにより「設計書には書いてあるが実際は動かない」問題を根絶

### Phase 3: タスク分解

**従来の対応物**: 詳細設計工程 + WBS作成

**J-SIXでの違い**:
- 従来の「詳細設計書」の代わりに、CCが実行可能な粒度のタスクに分解
- 各タスクに受入条件（テスト条件）を定義
- 依存関係を分析し、並列実行可能なタスクを特定

**品質担保**:
- タスクと受入条件を人間が承認
- ここが「CC自律実行圏」への最後のゲート

### Phase 4: TDD実装

**従来の対応物**: 実装 + 単体テスト

**J-SIXでの根本的な違い**:
- **テストを先に書く**（TDD）のが最も品質安定するパターン
- サブエージェント分離により、テスト作成者と実装者の「コンテキスト汚染」を防止
- Writer/Reviewer パターン：コードを書いたCCとは別のCCインスタンスがレビュー（自分のコードへのバイアスを排除）
- git worktree で複数タスクをCC並列実行

**品質担保**:
- TDD自体が品質保証メカニズム（テストが通らない限り実装は完了しない）
- CC相互レビュー（Writer/Reviewer分離）
- 自動回帰テストによるデグレード防止

### Phase 5: 品質検証

**従来の対応物**: 結合テスト + 総合テスト + 品質管理

**J-SIXでの違い**:
- テストケースをSpecから自動生成
- 品質メトリクスの自動収集・可視化
- 人間は「全数検証」ではなく「サンプリング検証」に移行

**品質担保**:
- **定量基準は従来と同等を維持**（バグ密度、カバレッジ率等）
- 自動化により網羅性はむしろ向上
- 人間は設計判断・業務ロジックの妥当性に集中

### Phase 6: ドキュメント生成

**従来の対応物**: （各工程で作成していた設計書を、ここで一括生成）

**J-SIXでの最大の革新**:
- 実装済みコードから設計書を **逆生成**
- 従来フォーマット（Excel/Word）での出力も対応可能
- コードが Source of Truth なので、設計書とコードの乖離が **原理的に発生しない**

**品質担保**:
- 生成された設計書を人間がレビュー
- 顧客に納品する設計書のフォーマット・内容は従来互換

## 5. J-SIXプロセスにおける人間の役割の再定義

### 5.1 人間が担う不可替の役割

| 役割 | 内容 | 従来の対応物 |
|---|---|---|
| **ビジネス翻訳者** | ステークホルダーの要求をSpecに翻訳 | 要件定義担当SE |
| **アーキテクト** | 技術的判断・トレードオフの決定 | アーキテクト |
| **品質の門番** | Phase Gateでの承認・判定 | PM/PL |
| **CLAUDE.md設計者** | プロジェクト憲法の策定・進化 | 開発標準策定者 |
| **サンプリング検証者** | CC出力の妥当性をサンプリングで確認 | レビュアー |
| **顧客折衝者** | 顧客との合意形成 | PM |

### 5.2 SE/PGの役割変化

```
【従来】
  SE: 設計書を書く人  →  PG: コードを書く人

【J-SIX】
  SE/PG の統合 → 「AI開発オーケストレーター」
    - Spec を書き、CC に指示を出す
    - CC の出力をレビューし、判断する
    - CLAUDE.md を育てる
    - 品質メトリクスを監視する
```

## 6. 日本特有の課題への対応

### 6.1 設計書文化との共存

**問題**: 日本のSI案件では、基本設計書・詳細設計書・IF設計書・DB設計書等の納品が契約で求められることが多い。

**対策**: Phase 6でコードから設計書を逆生成。フォーマットは従来互換。

**追加の利点**: コードと設計書が常に一致するため、従来発生していた「設計書とコードの乖離」問題が解消。

### 6.2 受発注関係・多重下請け構造

**問題**: 日本のSIは元請け→一次請け→二次請け→…の構造が一般的で、工程ごとの分業が前提。

**対策**:
- CLAUDE.mdにより、プロジェクトルールが全参加者（人間・CC）で統一
- 工程分業ではなく、Phase単位での分業に移行
- 下請け構造を前提とする場合も、各社がCCを活用することで品質を底上げ

### 6.3 品質保証部門（QA）の位置づけ

**問題**: 日本では独立したQA部門が品質ゲートを管理することが多い。

**対策**:
- QAの役割は「テスト実行」から「テスト戦略策定 + サンプリング検証」に進化
- 自動テストの網羅性・品質基準の設定がQAの主業務
- 探索的テスト（人間の直感によるテスト）の重要性はむしろ増大

### 6.4 IPA/SEC共通フレーム・PMBOK等との整合

**問題**: 共通フレームやPMBOKに準拠したプロセスが求められるケースがある。

**対策**:
- J-SIXのPhaseは共通フレームのアクティビティにマッピング可能
- Phase 6で生成する設計書により、監査証跡も従来互換で提供
- 「プロセスの変更」ではなく「プロセスの自動化」としてフレーミング

## 7. 議論すべきポイント（Open Questions）

以下は、J-SIXプロセスを確立するために議論が必要な論点である。

### 7.1 プロセスの粒度

- Phase 2（技術設計）と Phase 3（タスク分解）を分ける必要があるか？

- コードから逆生成できない設計情報（設計意図・却下した代替案等）はどう扱うか？
- ADR（Architecture Decision Records）を別途管理すべきか？

### 7.3 CC自律実行の範囲

- Phase 4 で CC を完全自律実行させるか、タスクごとに人間承認を入れるか？

### 7.4 既存フレームワークの採用 vs 独自設計

- BMAD、Superpowers 等の既存フレームワークをベースにすべきか？
- 日本特有の要件を考慮した独自プロセスを設計すべきか？
- あるいはハイブリッド（既存フレームワーク + 日本品質レイヤー）？

### 7.5 段階的移行の現実性

- 既存のV字モデル案件を、どの程度の期間でJ-SIXに移行できるか？
- 「V字モデル + CC支援」（前版）→「J-SIX」への段階的パスは？

### 7.6 セキュリティ・ガバナンス

- CC にどの範囲のコード・データへのアクセスを許可するか？
- 機密情報を含むプロジェクトでの CC 活用方針は？
