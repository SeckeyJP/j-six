# J-SIX レガシーコード適用ガイド

**Author**: H.Sekita | **Date**: 2026-03-29

> J-SIX は新規開発だけでなく、既存のレガシーコードベースにも段階的に適用できる。
> 本ガイドでは、レガシーコードに J-SIX を導入する際の戦略と注意点を示す。

---

## 1. レガシーコードベースの課題

J-SIX の Phase 4（TDD 実装）は「テストを先に書く」ことが前提だが、レガシーコードには以下の課題がある。

| 課題 | 影響 |
|---|---|
| テストがない or 少ない | TDD サイクルの起点がない |
| コードが密結合 | テスト可能な単位に分離できない |
| 設計書とコードが乖離 | Spec の作成が困難 |
| 暗黙のビジネスルール | コードを読んでも仕様が分からない |
| CC のコンテキスト制約 | 大規模コードベースは CC のコンテキストウィンドウに収まらない |

---

## 2. レガシーコードへの適用戦略

### 2.1 基本方針：「全面刷新」ではなく「島を作る」

レガシーコードベース全体を一度に J-SIX 化するのは非現実的。代わりに、レガシーの海の中に **J-SIX の島** を作り、島を徐々に拡大する。

```
レガシーコードベース（テストなし、密結合）
┌─────────────────────────────────────────────────┐
│                                                 │
│   ┌───────────┐                                 │
│   │ J-SIX島   │←── 新規モジュールはここに作る    │
│   │(TDD, Spec)│                                 │
│   └─────┬─────┘                                 │
│         │ Anti-Corruption Layer（腐敗防止層）    │
│         │                                       │
│   レガシーコード                                 │
│                                                 │
└─────────────────────────────────────────────────┘
```

**Anti-Corruption Layer（腐敗防止層）**: J-SIX 島とレガシーコードの間にインターフェースを置き、レガシーの設計がJ-SIX島に侵食するのを防ぐ。

### 2.2 4ステップの適用手順

```
Step 1: 理解          CC でレガシーコードを読み解く
Step 2: テストで囲む  変更対象の周囲にテストを書く
Step 3: 島を作る      新規コードは J-SIX（TDD）で書く
Step 4: 島を広げる    リファクタリング時にレガシーを J-SIX 化
```

---

## 3. 各ステップの詳細

### Step 1: CC でレガシーコードを理解する（自律度 L1）

CC はレガシーコードの読解に非常に有効。人間が何日もかけて読むコードを CC が短時間で分析できる。

**CC への指示例**:
```
このプロジェクトのコードベースを分析してください。
以下を調査してレポートにまとめてください:

1. アーキテクチャの概要（主要モジュールとその依存関係）
2. 技術スタック（言語、フレームワーク、ライブラリ）
3. データベース構造（テーブル一覧と主要な関連）
4. 外部連携先の一覧
5. テストの現状（カバレッジ、テストフレームワーク）
6. 暗黙のビジネスルール（コードから読み取れる業務ロジック）
```

**この段階のアウトプット**:
- コードベース分析レポート（CC 生成、人間がレビュー）
- CLAUDE.md の初版（技術スタック、ビルドコマンド等を記載）
- 暗黙のビジネスルールの洗い出し → ADR の遡及作成

> **ADR の遡及作成**: 過去の設計判断を CC がコードから推測し、ADR として記録する。「なぜこうなっているか」を CC に推測させ、当時の関係者に確認する。

### Step 2: 変更対象の周囲にテストを書く（自律度 L2）

Michael Feathers の「レガシーコード改善ガイド」の手法を CC で加速する。変更したい箇所の周囲にまず **特性化テスト（Characterization Test）** を書く。

**特性化テストとは**: コードの「あるべき振る舞い」ではなく「現在の振る舞い」を記録するテスト。リファクタリング時にデグレードを検出するための安全ネット。

**CC への指示例**:
```
src/services/order-service.js の createOrder 関数について、
特性化テストを書いてください。

特性化テストとは:
- 現在の振る舞いをそのまま記録するテスト
- 「正しい振る舞い」ではなく「現在の振る舞い」をアサート
- リファクタリング前の安全ネットとして機能する

以下のケースをカバーしてください:
- 正常系（標準的な注文）
- 境界値（数量0、金額0、最大値）
- 異常系（在庫不足、無効な商品ID）
```

**この段階のポイント**:
- CC は L2（人間が毎回レビュー）で運用。レガシーコードの理解が不十分な段階で L3 にしない
- テストが書けない（密結合すぎる）箇所を記録。Step 4 のリファクタリング対象リストになる
- テストフレームワークが未導入の場合は、この段階で導入

### Step 3: 新規コードは J-SIX（TDD）で書く（自律度 L3）

新規機能の追加・既存機能の大幅改修は、J-SIX の Phase 1-4 で行う。

**実装パターン**:

```
【パターンA: 新規モジュールの追加】
  レガシーとは独立した新モジュールを J-SIX で作成。
  Anti-Corruption Layer を介してレガシーと連携。

【パターンB: 既存機能の置き換え（Strangler Fig パターン）】
  レガシーの機能を新しい J-SIX 実装で徐々に置き換える。
  1. 新実装を J-SIX で作成
  2. ルーティング/フラグで新旧を切り替え
  3. 新実装の安定を確認後、旧実装を削除

【パターンC: 既存機能の改修】
  Step 2 のテストで囲んだ上で、内部実装を改修。
  改修部分は TDD で行う。
```

### Step 4: リファクタリング時にレガシーを J-SIX 化（自律度 L2-L3）

技術的負債の返済タスクとして、レガシーコードを段階的に J-SIX のコード品質基準に引き上げる。

**CC への指示例**:
```
src/services/order-service.js をリファクタリングしてください。

手順:
1. まず既存のテスト（Step 2 で追加済み）が全て通ることを確認
2. 以下の改善を1つずつ行い、各改善後にテストが通ることを確認
   - 関数の分割（30行以内に）
   - 命名規則の統一（CLAUDE.md のコーディング規約に準拠）
   - エラーハンドリングの統一
   - 型定義の追加（TypeScript 化する場合）
3. 各改善でコミットを分ける
4. 設計判断が発生したら ADR を作成
```

## 4. コンテキストウィンドウの制約への対処

レガシーコードベースが大規模な場合、CC のコンテキストウィンドウ（200Kトークン）に収まらない。

### 対処法

| 手法 | 内容 |
|---|---|
| **CLAUDE.md にコードマップを記述** | 主要モジュールの役割と依存関係を CLAUDE.md に簡潔に記述。CC がコードを読まなくても全体像を把握できるように |
| **モジュール単位で作業** | 一度に1モジュールずつ作業。コンテキストにはそのモジュールだけをロード |
| **サブエージェントで分業** | 調査（Explore Agent）と実装（Task Agent）を分離。調査結果のサマリーだけを実装 Agent に渡す |
| **.claudeignore の活用** | CC に読ませる必要のないファイル（ビルド成果物、node_modules 等）を除外 |
| **定期的な /compact** | コンテキストが 70% を超えたら /compact で圧縮。重要な情報は CLAUDE.md に永続化 |

## 6. 注意事項

### やってはいけないこと

- **レガシーコード全体を一度に CC にリファクタリングさせる**: コンテキスト不足で品質が低下する
- **特性化テストなしでリファクタリングする**: デグレードを検出できない
- **レガシーの設計を J-SIX 島に持ち込む**: Anti-Corruption Layer で分離すること
- **CC の自律度を L3 にいきなり上げる**: レガシーコードは CC の理解が不十分な領域。まず L1-L2 で信頼を構築

### レガシーコードで CC が特に有効な場面

- **コードの読解・ドキュメント生成**: 人間が数日かかる分析を数時間で
- **特性化テストの生成**: 既存の振る舞いを網羅的にテスト化
- **定型的なリファクタリング**: 命名統一、型追加、エラーハンドリング統一等
- **ADR の遡及作成**: 過去の設計判断をコードから推測して記録
