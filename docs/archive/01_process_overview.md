## 1. 本ドキュメントの位置づけ

本ドキュメントは、日本のSI業界で広く採用されているウォーターフォール型（V字モデル）の開発プロセスを前提に、Claude Code（以下CC）を各工程にどう組み込むかを俯瞰的に示すものである。

**対象読者**: PM・PL（プロセス導入判断者）、経営層・意思決定者
**前提**: 既存のプロセス・品質基準を「壊す」のではなく「強化・加速する」アプローチ

### 1.1 なぜ今、Claude Code なのか

日本のシステム開発現場が抱える構造的課題に対し、CCは以下の価値を提供する。

| 構造的課題 | CC がもたらす価値 |
|---|---|
| 慢性的な人材不足・高齢化 | シニアSEの暗黙知をCLAUDE.mdに形式知化し、チーム全体の底上げ |
| 設計書作成の工数肥大 | コードから設計書を逆生成、双方向トレーサビリティの自動維持 |
| 品質担保のためのレビュー負荷 | 静的解析・コード規約チェックの自動化でレビューを本質的議論に集中 |
| 下流工程での手戻りコスト | 上流段階でのプロトタイプ生成による早期フィードバック |
| 属人化したノウハウ | プロジェクト固有のルール・判断基準をCLAUDE.mdとして蓄積・継承 |

## 3. 工程別：何が変わり、何が変わらないか

### 凡例
- 🔵 **変わらない**（人間が責任を持つ領域）
- 🟢 **CCで加速**（人間が判断、CCが作業を支援）
- 🟡 **CCで変革**（プロセス自体が変わる）

### 3.1 要件定義

| 観点 | 従来 | CC活用 | 変化度 |
|---|---|---|---|
| ステークホルダーヒアリング | 対面・議事録ベース | 🔵 変わらない（人間の領域） | — |
| 要件の文書化 | Word/Excel手作業 | 🟢 議事録→要件定義書ドラフトをCC生成 | 加速 |
| 要件の曖昧性・矛盾検出 | レビュー会議で発見 | 🟡 CCが自動検出・指摘 | 変革 |
| 業務フロー作成 | Visio等で手描き | 🟢 テキスト記述→フロー図自動生成 | 加速 |
| 画面イメージ | モック作成に数日 | 🟡 要件からプロトタイプを即時生成 | 変革 |
| **品質ゲート（承認）** | **顧客レビュー・承認** | **🔵 変わらない** | **—** |

### 3.2 基本設計（BD）

| 観点 | 従来 | CC活用 | 変化度 |
|---|---|---|---|
| アーキテクチャ設計 | アーキテクトが策定 | 🔵 変わらない（人間が判断） | — |
| 画面設計書 | Excel方眼紙 | 🟡 実動HTMLモック＋仕様書自動生成 | 変革 |
| DB設計（ER図・テーブル定義） | ER図ツール＋Excel | 🟢 テーブル定義→DDL→ER図を一貫生成 | 加速 |
| API/IF設計書 | Excel手作業 | 🟢 OpenAPI Spec自動生成→IF設計書変換 | 加速 |
| 非機能要件定義 | チェックリストベース | 🔵 変わらない（要件による） | — |
| **品質ゲート（設計レビュー）** | **レビュー会議** | **🟢 CC事前チェック＋人間レビュー** | **加速** |

### 3.3 詳細設計（DD）

| 観点 | 従来 | CC活用 | 変化度 |
|---|---|---|---|
| クラス図・シーケンス図 | 手作業で作成 | 🟡 コード骨格と設計図を同時生成 | 変革 |
| 処理フロー記述 | 日本語で記述 | 🟡 設計記述→コード骨格の双方向変換 | 変革 |
| 入出力設計 | Excel定義 | 🟢 バリデーション仕様→実装を同時生成 | 加速 |
| 設計書⇔コード整合性 | 目視確認 | 🟡 自動整合性チェック | 変革 |
| **品質ゲート（設計レビュー）** | **レビュー会議** | **🟢 CC事前チェック＋人間レビュー** | **加速** |

### 3.4 実装（コーディング）

| 観点 | 従来 | CC活用 | 変化度 |
|---|---|---|---|
| コーディング | 開発者が全行記述 | 🟡 CC生成＋開発者レビュー・修正 | 変革 |
| コーディング規約遵守 | Linter＋目視 | 🟢 CLAUDE.mdで規約定義→自動適用 | 加速 |
| コードレビュー | ペアレビュー | 🟢 CC事前レビュー＋人間レビュー | 加速 |
| **品質ゲート（コードレビュー）** | **レビュー会議** | **🟢 二段階レビュー（CC→人間）** | **加速** |

### 3.5 テスト（UT/IT/ST）

| 観点 | 従来 | CC活用 | 変化度 |
|---|---|---|---|
| テストケース作成 | 設計書から手作業 | 🟡 設計書＋コードから自動生成 | 変革 |
| テストコード作成 | 開発者が記述 | 🟡 CC自動生成＋開発者レビュー | 変革 |
| テスト実行 | CI/CD | 🔵 変わらない | — |
| カバレッジ分析 | ツールで計測 | 🟢 不足箇所のテスト自動追加 | 加速 |
| テストエビデンス作成 | スクリーンショット貼り | 🟢 エビデンス収集・整理を自動化 | 加速 |
| **品質ゲート（テスト完了判定）** | **バグ密度・カバレッジ基準** | **🔵 基準は変わらない** | **—** |

## 4. CLAUDE.md：プロジェクト統制の要

CC活用プロセスにおいて最も重要な成果物が **CLAUDE.md** である。これは従来の「開発標準」「コーディング規約」「プロジェクト計画書」に相当する情報をCCが理解・実行可能な形式で記述したものである。

### 4.1 CLAUDE.md に定義すべき内容

```
CLAUDE.md
├── プロジェクト概要
│   ├── システムの目的・業務概要
│   ├── アーキテクチャ方針
│   └── 技術スタック
├── 開発規約
│   ├── コーディング規約（命名規則、フォーマット）
│   ├── ディレクトリ構成ルール
│   ├── Git運用ルール（ブランチ戦略、コミットメッセージ）
│   └── エラーハンドリング方針
├── 設計方針
│   ├── DB設計ルール（命名規則、正規化方針）
│   ├── API設計ルール（RESTful規約、認証方式）
│   └── 画面設計ルール（UI/UXガイドライン）
├── 品質基準
│   ├── テストカバレッジ目標
│   ├── 静的解析ルール
│   └── パフォーマンス基準
└── 禁止事項・制約
    ├── 使用禁止ライブラリ
    ├── セキュリティ制約
    └── 本番環境に関する制約
```

### 4.2 従来成果物との対応関係

| 従来の成果物 | CLAUDE.md での対応 | 備考 |
|---|---|---|
| プロジェクト計画書 | プロジェクト概要セクション | スコープ・方針のみ |
| 開発標準・規約書 | 開発規約セクション | CCが自動適用 |
| コーディング規約 | 開発規約 > コーディング規約 | 違反を自動検出・修正 |
| 設計ガイドライン | 設計方針セクション | 設計書生成時に自動反映 |
| テスト計画書 | 品質基準セクション | テスト生成の基準として機能 |

## 5. 品質保証の考え方：変わるもの・変わらないもの

### 5.1 変わらないもの（堅持すべき原則）

1. **品質ゲートの存在** — 各工程完了時の品質判定基準と承認プロセス
2. **品質メトリクス** — バグ密度、テストカバレッジ、レビュー指摘密度等の定量管理
3. **トレーサビリティ** — 要件→設計→実装→テストの追跡可能性
4. **顧客承認プロセス** — 要件定義・基本設計の顧客合意
5. **第三者検証** — QAチームによる独立したテスト実施

### 5.2 変わるもの（進化する領域）

1. **レビューの質的転換** — 形式チェックはCCに委ね、人間は設計判断・業務妥当性に集中
2. **シフトレフトの実現** — 上流工程でのプロトタイプ・コード生成による早期検証
3. **継続的整合性チェック** — 設計書⇔コードの乖離を常時監視
4. **テスト網羅性の向上** — 境界値・異常系の自動網羅による品質底上げ
5. **ナレッジの形式知化** — 暗黙知だった判断基準がCLAUDE.mdとして可視化・継承

## 6. 導入ロードマップ（段階的移行）

### Phase 1：Quick Win（1〜2ヶ月）
**対象工程**: 実装・単体テスト
**施策**:
- CLAUDE.md の初期整備（コーディング規約・ディレクトリ構成）
- コード生成・テストコード生成の試行
- 既存プロジェクトの一部モジュールで効果検証

**期待効果**: 実装工数 30〜50% 削減、テストカバレッジ向上

### Phase 2：設計工程への拡大（3〜4ヶ月）
**対象工程**: 詳細設計・結合テスト
**施策**:
- 設計書テンプレートのCC対応化
- 設計書⇔コードの双方向生成フロー確立
- 結合テストケース自動生成

**期待効果**: 設計書作成工数 40〜60% 削減、設計⇔実装の整合性向上

### Phase 3：上流工程への拡大（5〜6ヶ月）
**対象工程**: 要件定義・基本設計
**施策**:
- 要件の曖昧性自動検出
- プロトタイプ駆動の要件確認フロー
- 基本設計書の自動生成

**期待効果**: 手戻り削減、顧客合意形成の迅速化

### Phase 4：プロセス全体最適化（7ヶ月〜）
**対象**: 全工程横断
**施策**:
- 全工程でのCC活用の標準化
- 品質メトリクスの自動収集・可視化
- プロジェクト横断でのCLAUDE.mdベストプラクティス共有

**期待効果**: 開発プロセス全体の生産性 2〜3倍向上

## 7. リスクと対策

| リスク | 影響度 | 対策 |
|---|---|---|
| CC生成コードの品質ばらつき | 高 | CLAUDE.mdでの厳密なルール定義＋人間レビュー必須化 |
| 開発者のスキル低下懸念 | 中 | CC出力のレビュー能力を重視した人材育成に転換 |
| セキュリティ（コード外部送信） | 高 | オンプレ/VPC環境での利用、機密情報のマスキングルール |
| 既存プロセスとの軋轢 | 中 | 段階的導入＋既存品質基準は維持する方針の明示 |
| CC依存によるベンダーロック | 中 | CLAUDE.mdの記述をベンダー非依存な形式で設計 |
| 顧客・監査への説明責任 | 高 | AI活用ガイドラインの策定、品質エビデンスの従来形式での出力 |
