## 2. 概要編（#0）の詳細構成

### タイトル案

**「J-SIX — Claude Code で日本のSI開発プロセスを再定義する」**

副題: V字モデルを捨てるのではなく、進化させる。AI ネイティブ開発プロセスの提案

### ターゲット

- **PM・PL**: 「うちのプロジェクトでも使えるか？」の判断材料
- **経営層**: 「投資対効果はあるか？」の評価材料
- **SE**: 全体像を掴んでからシリーズを読む入口

### 想定ボリューム

Qiita/Zenn で **読了8-12分**（約4,000-6,000字）

### 構成

```
■ リード（200字）
  日本のSI業界で主流のV字モデル。30年以上使われてきたこのプロセスの
  前提が、Claude Code の登場で根本から崩れつつある。
  本記事では、Claude Code をフル活用した AI ネイティブ開発プロセス
  「J-SIX」を提案する。

■ 1. V字モデルの前提が崩壊している（800字）
  ・「実装コストが高い」→ CC は数分で数百行を生成
  ・「テスト作成が高コスト」→ テストコード自動生成
  ・しかし「何を作るか」「品質の最終責任」は変わらない
  ・CC の現実的な能力水準（人間介入33%減少、イシュー率1.7倍）
    ※データは出典付きで提示

■ 2. 世界では何が起きているか — SDD の台頭（600字）
  ・Spec-Driven Development とは何か
  ・BMAD、Superpowers 等のフレームワークの登場
  ・共通原則: Spec First / Phase Gate / TDD / 並列エージェント
  ※ 詳細はシリーズ #1 で

■ 3. J-SIX の4原則（800字）
  ① CC ネイティブ — 外部FWに依存しない
  ② 3層ドキュメント — Spec + ADR + 逆生成設計書
  ③ TDD が品質の中核 — テストファーストで自律実行の品質を担保
  ④ 段階的移行 — 既存案件を止めない3ステージ移行
  ※ 図解: Phase 0-6 の全体像

■ 4. 最大のパラダイムシフト：設計書は「逆生成」(800字)
  ・従来: 設計書→実装→（乖離発覚）→手戻り
  ・J-SIX: Spec→テスト→実装→検証→設計書の逆生成
  ・コードが Source of Truth なので乖離が原理的にゼロ
  ・「設計書がなくなる」のではなく「作り方が変わる」
  ・ADR で「Why / Why Not」を補完
  ※ 詳細はシリーズ #2 で

■ 5. 段階的に移行できる（600字）
  ・Stage 1（3ヶ月）: V字+CC補助。工数25%削減
  ・Stage 2（3-6ヶ月）: SDD要素を導入。工数45%削減
  ・Stage 3（6ヶ月〜）: J-SIX 全面適用。工数65%削減
  ・ROI試算: CC月額7.5万円 vs 削減効果月100万円〜
  ※ 数値は著者推定であることを明記
  ※ 詳細はシリーズ #5 で

■ 6. 明日からできること（600字）
  ・Step 1: CLAUDE.md を書く（テンプレート配布中）
  ・Step 2: 実装+UT で CC を使い始める
  ・Step 3: 効果を計測する
  ・GitHub リポジトリへのリンク（テンプレート・ガイド一式）

■ まとめ（200字）
  ・J-SIX は V字モデルを「捨てる」のではなく「進化させる」
  ・品質基準は維持。変わるのは「作り方」
  ・シリーズ記事の案内

■ 参考文献
  ・記事内で引用した出典を列挙
  ・GitHub リポジトリ URL
```

## 4. 記事共通の方針

### トーン

- **押し付けない**: 「こうすべき」ではなく「こういう選択肢がある」
- **正直に限界を示す**: CC の能力と限界を出典付きで提示、推定値は推定と明記
- **実践志向**: 概念だけでなく「明日からできること」を必ず含める
- **出典を示す**: 主要データは必ず出典付き。著者推定は明記

### フォーマット

- Qiita / Zenn 両方に投稿（内容は同一）
- シリーズタグ: `j-six` `claude-code` `ai-development` `si`
- 各記事の冒頭にシリーズ一覧のリンク
- 各記事の末尾に GitHub リポジトリへのリンク

### 図解

- 各記事に最低1つの図解を含める
- Mermaid or テキストベース図（Qiita/Zenn で表示可能な形式）
- J-SIX.md の図をベースに記事用に簡略化
